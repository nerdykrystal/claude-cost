## 2. Functional Requirements

### 2.1 Core Functional Requirements

**FR-01: Parse D2R plan documents**
- Statement: The system MUST accept a D2R plan document in YAML or Markdown format and parse it into a validated internal representation per the formal Zod schema.
- Rationale: D2R plans are the primary input (PRD Journey 1, 2, 3)
- Acceptance: given a well-formed D2R plan, parser produces valid internal representation with zero errors; given a malformed plan, parser produces specific parse errors pointing to problem locations
- Priority: Must-Have

**FR-02: Calculate per-stage cost estimate**
- Statement: The system MUST calculate the cost of each stage in a parsed D2R plan given the stage's assigned model, effort level, and context-carrying configuration.
- Rationale: Core estimation capability (PRD Problem 1)
- Acceptance: calculated cost matches hand-verified expected value within floating-point tolerance for a test corpus of 20 known-input plans
- Priority: Must-Have

**FR-03: Calculate total plan cost estimate**
- Statement: The system MUST produce a total plan cost by rolling up per-stage costs with context-carrying accumulation, caching discounts, batch discounts, thinking-mode ceilings, and tokenizer corrections applied.
- Rationale: Core estimation capability
- Acceptance: rolled-up total matches expected for test corpus
- Priority: Must-Have

**FR-04: Apply Opus 4.7 tokenizer correction**
- Statement: The system MUST apply a tokenizer correction factor for any stage using Opus 4.7 or later, and surface the correction to the user with a visible explanation whenever a plan contains mixed Opus generations.
- Rationale: Critical correctness risk per tech stack research
- Acceptance: tokenizer correction factor is empirically measured and documented; mixed-generation plans display visible callout; calculations include correction
- Priority: Must-Have

**FR-05: Calculate retry scenario envelope**
- Statement: The system MUST calculate best-case / expected / worst-case cost scenarios accounting for retry probability per ASAE gate stage.
- Rationale: PRD Journey 1 requires envelope, not point estimate
- Acceptance: given test plans with known retry configurations, envelope bounds match expected values
- Priority: Must-Have

**FR-06: Compare multiple plans**
- Statement: The system MUST accept 2 to 20 D2R plans simultaneously and produce comparison views (totals, per-stage deltas, cost concentration heatmap).
- Rationale: PRD Journey 2
- Acceptance: comparison of N plans displays all N costs side-by-side; heatmap highlights where cost concentrates differently; deltas computed accurately
- Priority: Must-Have

**FR-07: Post-run variance analysis**
- Statement: The system MUST accept execution log data (actual token counts, actual retries, actual cache hits/misses, actual model used per stage) and produce variance analysis against prior estimates.
- Rationale: PRD Problem 2, PRD Journey 3
- Acceptance: given paired estimate + actual data, system produces per-stage variance, attribution to specific factors, and aggregate variance report
- Priority: Must-Have

**FR-08: Export results**
- Statement: The system MUST export estimation and analysis results in CSV, JSON, and Markdown report formats.
- Rationale: PRD Journey 4 and general usability
- Acceptance: each format produces valid output of the full dataset; formats are round-trippable where semantically possible
- Priority: Must-Have

**FR-09: Pricing data with freshness indicator**
- Statement: The system MUST display the date of the bundled pricing data and show a visible warning if the data is older than 30 days.
- Rationale: Correctness risk — pricing changes without notice
- Acceptance: pricing data date always visible; >30 day warning triggers appropriately
- Priority: Must-Have

**FR-10: CLI wrapper for programmatic use**
- Statement: The system MUST provide a CLI that accepts a D2R plan file path and outputs cost estimates in JSON format for programmatic consumption.
- Rationale: Experimental workflow (running 11+ conditions)
- Acceptance: `claude-cost estimate plan.yaml --output cost.json` produces valid JSON output matching the web UI's calculation
- Priority: Must-Have

### 2.2 User-Facing Behavior Requirements

**BR-01: Plan upload via file selector**
- Input: user clicks "upload plan" → selects YAML/Markdown file
- Expected output: plan parsed, cost estimate displayed within 2s for plans ≤ 100KB
- Timing: synchronous from user perspective

**BR-02: Plan paste via text area**
- Input: user pastes YAML/Markdown text into text area → clicks "parse"
- Expected output: plan parsed, estimate displayed within 2s
- Timing: synchronous

**BR-03: Multi-plan comparison**
- Input: user uploads/pastes 2+ plans
- Expected output: comparison view with all plans, deltas, heatmap
- Timing: synchronous for ≤ 5 plans; may show progress indicator for 6-20

**BR-04: Post-run log upload**
- Input: user uploads execution log (JSON format with stage-level metrics)
- Expected output: variance analysis displayed paired with prior estimate
- Timing: synchronous

**BR-05: Export action**
- Input: user clicks "export" on any view → selects format
- Expected output: file download begins within 1s; file contains complete dataset
- Timing: synchronous for CSV/JSON; may show progress for Markdown if large

### 2.3 System-Facing Behavior Requirements

- Pricing data bundled at build time from `@pydantic/genai-prices` + custom Anthropic advanced-pricing overlay
- No background jobs, scheduled tasks, or external API calls at runtime
- All data processing happens in the user's browser or CLI process

## 3. Non-Functional Requirements

### 3.1 Performance Requirements

- Page load: LCP ≤ 2.5s, FID ≤ 100ms, CLS ≤ 0.1 (Core Web Vitals thresholds)
- Plan parse: ≤ 200ms for plans ≤ 100KB
- Cost estimation: ≤ 500ms for single plan of 20 stages
- Multi-plan comparison: ≤ 2s for 11-plan factorial
- Bundle size: ≤ 250KB gzipped for initial load (excludes Pyodide which lazy-loads)
- Concurrent usage: N/A — pure client-side, each user runs independently
- Degradation: plans > 100KB show progress indicator; plans > 1MB display size warning with recommendation to split

### 3.2 Reliability Requirements

- Uptime: 99.9% (inherits from Vercel / Cloudflare Pages SLA)
- MTBF: N/A for pure static deployment
- RTO: immediate (static redeploy)
- RPO: zero (no runtime state; plans not persisted to any server)
- Failure modes tolerated: network loss after initial load (tool continues working client-side); pricing data up to 30 days stale (warning surfaces)
- Failure modes prohibited: silent incorrect calculations, data loss during export

### 3.3 Security Requirements

- Authentication: none (public tool; no user accounts)
- Authorization: none (all features available to all users)
- Data in transit: HTTPS only; HSTS with preload eligibility
- Data at rest: none (no server-side storage)
- Secret management: no secrets in product; pricing data is public
- Audit logging: none (no server to log; client-side only)
- Applicable standards:
  - OWASP Top 10 Web Application Security Risks (2021 version): applicable items verified (A03 Injection via plan parsing, A05 Security Misconfiguration via CSP, A06 Vulnerable Components via dep audit)
  - CERT Secure Coding Standards for TypeScript / JavaScript: applicable
  - OWASP LLM Top 10: NOT applicable (no LLM integration in product itself)
- Threat model summary: primary threat is malicious plan input causing XSS or CSP violation during rendering. Mitigation: strict CSP, all user input escaped before render, no dynamic eval of plan content.

### 3.4 Privacy Requirements

- Personal data collected: none (tool is a calculator)
- Applicable regulations: none apply to tool operation; GDPR/CCPA require no accommodation since no personal data processed
- Data minimization: zero data collected
- Retention: zero (no server-side storage)
- User data rights: N/A (no data to access / delete / port)
- Cross-border transfer: N/A

### 3.5 Accessibility Requirements

- Target compliance: **WCAG 2.1 AA (hardwired minimum)**
- Additional standards: AAA criteria adopted where not adding disproportionate complexity; specifically: 1.4.6 Contrast (Enhanced) where practical, 2.3.3 Animation from Interactions (respect prefers-reduced-motion)
- Keyboard navigation: 100% of functionality operable without mouse
- Screen reader compatibility: tested combinations:
  - NVDA (latest) + Firefox (latest) on Windows
  - JAWS (latest) + Edge (latest) on Windows
  - VoiceOver (latest macOS) + Safari (latest)
- Color contrast: 4.5:1 minimum (normal text), 3:1 minimum (large text), 4.5:1 minimum for UI components per 1.4.11
- Motion: `prefers-reduced-motion` respected on all animations; no animation lasts longer than 5 seconds (2.2.2)
- Automated testing: axe-core via `@axe-core/playwright` in CI; zero `critical` or `serious` violations required
- Manual testing: documented protocol in TQCD; executed before each release

### 3.6 Maintainability Requirements

- Expected lifespan: 2+ years of active maintenance
- Maintainer: solo (Krystal) with AI-assisted execution
- Code quality gates:
  - Cyclomatic complexity: median ≤ 10, p95 ≤ 15
  - Line + branch coverage: 100% per D2R hardwired
  - ESLint strict config with zero warnings permitted
  - TypeScript strict mode (`"strict": true`) with zero type errors
- Documentation quality floor: every exported function has JSDoc; every module has a header comment explaining purpose; README covers setup + usage

### 3.7 Portability Requirements

- Target platforms:
  - Web: Firefox 115+, Chrome 120+, Safari 17+, Edge 120+
  - CLI: Node.js 20+ on Linux / macOS / Windows
  - Tauri desktop (v1): Windows 10+, macOS 11+, Linux (Ubuntu 22.04+)
- Target runtime: browser (web) or Node.js (CLI)
- Deployment: Vercel or Cloudflare Pages (web); npm (CLI)
- Bundle size: ≤ 250KB gzipped initial; Pyodide lazy-loaded (~15MB) only if user requests Jupyter export

### 3.8 Observability Requirements

- Logging: client-side console logging in development; no server logs (no server)
- Metrics: none in MVP (no analytics for privacy reasons); may add opt-in anonymous usage counters in v1
- Tracing: N/A
- Alerting: N/A

## 5. Data Requirements

### 5.1 Data Entities And Schema

- **D2RPlan** — structured YAML/Markdown representation of a multi-stage AI pipeline (Zod schema TBD in Stage 00)
- **PricingModel** — model name, input/output/thinking token prices, cache tier prices, batch discount, tokenizer correction factor
- **EstimateResult** — per-stage costs, total cost, envelope bounds, assumptions log
- **ExecutionLog** — post-run actuals (per-stage actual tokens, retries, cache outcomes, model used, duration)
- **VarianceReport** — estimated vs actual delta with attribution

### 5.2 Data Volume Expectations

- Typical plan: 5-20 stages
- Max plan: 100 stages (anything larger should be split)
- Typical comparison: 2-11 plans (11 matches experimental factorial)
- Max comparison: 20 plans
- No persistent storage; in-memory only

### 5.3 Data Sensitivity Classification

- D2R plans: internal / confidential (may contain proprietary pipeline descriptions) — processed locally only, never transmitted
- Pricing data: public
- Execution logs: may contain model outputs; treat as same sensitivity as user's source logs — processed locally only

### 5.4 Data Validation Requirements

- Plan validation: Zod schema enforced on parse; specific error messages for violations
- Schema validation approach: Zod for runtime + type inference
- Invalid data: reject with specific error; do not attempt sanitization of malformed plans

## 6. Technical Constraints

### 6.1 Mandatory Technology Choices

- **Pure client-side architecture** (no backend server) — per PRD privacy and simplicity goals
- **TypeScript** (no plain JavaScript) — per maintainability requirements
- **WCAG 2.1 AA compliance** — hardwired

### 6.2 Prohibited Technology Choices

- **Meta Llama as primary open-weight reference** — per Martinez Methods alignment-with-excellence principle; if open-weight modeling is added, use NVIDIA Nemotron or similar non-Meta options
- **React as primary UI framework** — 120KB+ baseline bundle conflicts with performance budget and audience constraints (confirmed via Stage 00 tech stack research)
- **Tracking / analytics SDKs** — privacy posture

### 6.3 Platform Target

- MVP: web static app + Node.js CLI wrapper
- v1: + Tauri v2 desktop + npm SDK + Claude Code plugin form
- Rationale: matches PRD user segments (non-coders via web, experimental users via CLI, privacy-sensitive users via desktop, integration users via SDK/plugin)

### 6.4 Hook Orchestration Requirements (D2R-Specific)

**Claude Code hooks** in `.claude/settings.json`:
- PreToolUse on `Write|Edit` in main session: block direct code writes, force delegation to assigned sub-agent per stage
- PreToolUse on `Bash` matching `git commit*`: verify ASAE summary for current stage in thread context before commit
- PreToolUse on `Bash` matching `git push*`: verify D2R final summary exists before push
- PostToolUse on `Write|Edit`: log to audit trail
- Stop / StopFailure: require ASAE summary before clean session end
- UserPromptSubmit: inject active stage state into context

**Git hooks** in `.githooks/`:
- `pre-commit`: format → lint → type check → test → axe-core (if UI) → ASAE log validation → secret scan
- `pre-push`: full regression suite + build

**ASAE thresholds per stage:**
- Stage 00 (research, Opus): 2 (research rigor)
- Stage 01a (skeleton authorship, Opus): 2 (rapid iteration — user approval is the deeper gate)
- Stage 01b (full plan authorship, Opus): 3 (plan-content completeness; Deep-spec stages must have all Deep-depth requirements present)
- Stage 02 (project scaffold, Sonnet): 3 (scaffold completeness; repo + docs + configs + hooks + CI installed and green)
- Stage 03-NN (feature implementation, Haiku): 3 (implementation correctness against Stage 01b Deep spec)
- Stage QA (convergence loop): 5 (terminal convergence)

**Severity policy:** strict (any CRITICAL, HIGH, or MEDIUM finding resets the counter)
