# Claude Cost — Product Requirements Document

## 1. Product Identity

### 1.1 Product Name

**Claude Cost.** Part of the Martinez Methods product portfolio. Referenced in the Martinez Methods Product Concept Register under the Claude \* Unblackboxing family as a measurement-infrastructure tool.

### 1.2 Version

**MVP v1.0** — first production release, scoped for Martinez Methods experimental infrastructure use and broader practitioner use.

### 1.3 One-Line Description

**Claude Cost estimates the cost of D2R code plans before execution and analyzes actual vs. estimated costs after execution, with cross-plan comparison and publication-ready export.**

---

## 2. Users And Problem

### 2.1 Primary Users

**Segment 1: Martinez Methods Experimental Researcher (Krystal specifically, plus collaborators).** Running the 11-condition D2R methodology factorial experiment. Needs to estimate per-condition costs pre-run, capture actual costs post-run, and produce statistically defensible comparisons. Currently: no tool exists to do this rigorously; would otherwise hand-roll in spreadsheets.

**Segment 2: AI-First Practitioners Planning Multi-Model Pipelines.** Non-enterprise practitioners (solopreneurs, small consultancies, researchers, NGO consultants) who design AI pipelines and need to estimate cost before committing to execution. Currently: estimate costs ad hoc from pricing pages, often wildly off because they don't account for caching, batching, thinking tokens, tokenizer differences, or context-carrying.

**Segment 3: Research Community Comparing AI Pipeline Approaches.** Researchers producing published comparisons of AI pipeline methodologies (including but not limited to Martinez Methods) who need a reproducible cost measurement instrument. Currently: each research group builds its own cost analysis; results are not directly comparable across papers.

### 2.2 Secondary Users

**Segment 4: WOC microbusinesses, small schools, and mission-driven orgs adopting AI.** Need to understand AI pipeline costs before committing budget. Typically non-coder. Benefit from accessible non-technical UI. This is the population Martinez Methods' open-source commitment is aimed at serving.

### 2.3 Problem Statement

Three distinct problems solved by one tool:

**Problem 1: Cost estimation for multi-model AI pipelines is currently unreliable.** Existing pricing calculators handle single-model single-call scenarios. They do not model: multi-stage pipelines with cross-stage context carrying, prompt caching discounts across cache tiers, batch API discounts, thinking-mode token accounting, tokenizer differences between model versions (specifically Opus 4.7's ~35% token overhead vs Opus 4.6 for identical text), retry scenarios from failed ASAE gates. A practitioner estimating a pipeline cost today will be materially wrong.

**Problem 2: Post-run cost analysis infrastructure doesn't exist.** After running a pipeline, practitioners have API response metadata with token counts but no systematic way to compare actual-vs-estimated, attribute variance to specific stages or decisions, or calibrate future estimates from past actuals. Every cost analysis is hand-rolled.

**Problem 3: AI pipeline research cannot produce directly comparable cost claims.** Published papers comparing AI pipeline approaches use inconsistent measurement methods. A paper claiming "method A is 10x cheaper than method B" is not reproducible because the measurement methodology isn't standardized. The research community needs a shared instrument.

### 2.4 Why Now

Three environmental changes make this the right moment:

1. **Opus 4.7 released April 16, 2026** with a new tokenizer that adds ~35% token overhead for identical text. This is the specific correctness problem that makes ad hoc estimation now materially misleading for anyone comparing across Opus generations.
2. **Multi-model AI pipelines are becoming standard practice.** The D2R + ASAE methodology specifically routes work across Opus/Sonnet/Haiku with plan-specified assignments. Cost modeling at this granularity is necessary for rational planning.
3. **AI pipeline research is scaling up.** MIT Sloan's 95% GenAI pilot failure statistic (2025) made cost measurement a field-level concern. Research evaluating methodologies against the failure rate needs reproducible cost instruments.

## 3. Goals

### 3.1 Primary Goals

**Goal 1: Enable the Martinez Methods 11-condition factorial experiment.**
- Measurable criterion: all 11 conditions have estimated + actual costs captured, variance analyzed per condition, comparative report exported
- Timeframe: available before first experimental condition runs

**Goal 2: Provide practitioners accurate cost estimates for multi-model AI pipelines.**
- Measurable criterion: estimates within ±20% of actual cost on typical pipelines (measured by post-run analysis), with all major variance factors (tokenizer differences, caching, batching, thinking-mode, retries) modeled
- Timeframe: from MVP release

**Goal 3: Establish a citable, reproducible cost measurement methodology for published research.**
- Measurable criterion: exported Jupyter notebook / Quarto document contains complete inputs, pricing snapshot, calculation source, and results — sufficient for independent reproduction
- Timeframe: v1.0 release

### 3.2 Secondary Goals

- Demonstrate Martinez Methods methodology on a tool that directly serves the methodology's evidence base
- Create a reference example of D2R + ASAE applied to a real build for future documentation
- Produce a measurement instrument that non-Anthropic AI researchers can also use, extending Martinez Methods reach

### 3.3 Non-Goals

- **Not a billing platform.** Claude Cost estimates and analyzes. It does not collect payments, issue invoices, or manage cost centers.
- **Not a cost optimizer.** Claude Cost measures. It does not automatically rewrite plans to minimize cost.
- **Not a runtime cost monitor.** Claude Cost analyzes structured plan documents and execution logs. It does not instrument running pipelines in real time.
- **Not an LLM provider-agnostic abstraction layer.** Claude Cost models the specific pricing of specific providers. It is not a general "cost" concept.
- **Not a replacement for D2R or ASAE.** Claude Cost is the measurement substrate. The planning and governance live in their respective skills.

## 4. User Journeys

### 4.1 Primary User Journeys

**Journey 1: Estimate a single D2R plan before execution**
- Goal: Understand cost of a plan before committing to running it
- Steps: User loads the tool → pastes or uploads a YAML/Markdown D2R plan → tool parses and validates → cost estimate displays (per-stage + total + best-case/expected/worst-case envelope) → user drills into stages to understand variance sources
- Outcome: User has a cost estimate they can trust to make go/no-go decisions
- Pain eliminated: ad hoc per-stage pricing-page lookups that miss caching/batching/tokenizer corrections

**Journey 2: Compare multiple D2R plans**
- Goal: Decide between alternative plan configurations based on cost delta
- Steps: User loads N plans → tool produces comparison view (per-plan totals, per-stage deltas, heatmap of cost concentration) → user identifies cost drivers and trade-offs → exports comparison for decision record
- Outcome: Rational plan selection based on quantified cost deltas
- Pain eliminated: subjective "this seems cheaper" reasoning

**Journey 3: Analyze actual vs estimated after execution**
- Goal: Understand what drove cost variance from estimates; calibrate future estimates
- Steps: User uploads execution log (actual token counts per stage + retries + caching outcomes) → tool pairs actuals with prior estimate → variance analysis report (where overrun happened, which stages + models + decisions drove it) → user exports report for audit trail
- Outcome: Closed-loop calibration infrastructure; learnings feed into future plans
- Pain eliminated: post-run "we spent a lot but don't know why" blindness

**Journey 4: Export for research publication**
- Goal: Produce reproducible cost analysis artifact for paper submission
- Steps: User runs estimates and post-run analysis on their experimental conditions → exports a Quarto / Jupyter reproducibility package → package contains plan inputs, pricing snapshot, calculation source, results → user attaches package to paper submission
- Outcome: Independently reproducible cost claims in published research
- Pain eliminated: cost claims in papers that can't be verified or reproduced

### 4.2 Edge Case Journeys

- **Journey 5: Handle a malformed plan.** User uploads a plan that doesn't match the Zod schema. Tool rejects with specific parse errors pointing to problem lines. User corrects plan and retries.
- **Journey 6: Handle stale pricing data.** User loads tool after >30 days since last pricing update. Tool displays warning with "pricing data from [date]; may be stale." User acknowledges or updates pricing data.
- **Journey 7: Handle Opus 4.7 tokenizer correction.** User's plan mixes Opus 4.7 and Opus 4.6 stages. Tool surfaces the tokenizer correction explicitly with a visible callout; calculations include the correction factor; export notes it.

## 5. Success Criteria

### 5.1 Measurable Outcomes

**Outcome 1: Experimental infrastructure ready**
- Metric: 11 Martinez Methods experimental conditions have documented pre-run estimates captured through Claude Cost
- Target: 11/11 before experiment begins
- Method: check against the experiment run log
- Frequency: once, pre-experiment

**Outcome 2: Estimate accuracy**
- Metric: absolute percentage error between estimate and actual cost for any single pipeline run
- Target: p50 ≤ 15%, p95 ≤ 30%
- Method: post-run analysis feature's own variance report
- Frequency: aggregated monthly across all runs

**Outcome 3: Reproducibility package validity**
- Metric: independent third party can re-run the analysis and reproduce the result using only the exported package
- Target: 100% of exported packages reproducible
- Method: one-time external audit at v1.0
- Frequency: once for v1.0 validation; spot-check thereafter

**Outcome 4: Accessibility compliance**
- Metric: axe-core violations at `critical` or `serious` level on any page
- Target: 0
- Method: axe-core via Playwright in CI
- Frequency: every commit

### 5.2 Qualitative Success Signals

- Martinez Methods experimental claims produced from Claude Cost outputs are not disputed on cost-methodology grounds by reviewers
- Practitioners outside Martinez Methods adopt the tool for their own pipeline planning
- Published research papers cite Claude Cost as the cost measurement instrument used

## 6. Constraints

### 6.1 Business Constraints

- **Zero-budget build.** This project ships without external funding. All infrastructure must be free-tier or minimal-cost (Vercel / Cloudflare Pages free tier acceptable).
- **Open-source commitment.** The tool is open source under a license protective of small users (specific license selection deferred to legal review pre-launch).
- **Non-coder maintainability.** The primary author is a non-coder architecting Opus-plans-Haiku-codes. Build complexity must be appropriate to that model — not requiring ongoing manual engineering intervention for operational reasons.

### 6.2 Regulatory Constraints

- No PII is collected or processed — the tool is a calculator operating on structured plan documents. GDPR / CCPA / COPPA not applicable to tool operation.
- If users later upload execution logs that may contain PII (unlikely given log content), the tool must process entirely client-side and never transmit logs to any server.

### 6.3 Technical Constraints

- **No Anthropic API key required to operate the tool.** The tool estimates from pricing data; it does not call any LLM API.
- **Pure client-side architecture for the web surface.** No backend server that sees user plan data.
- **Platform target:** web static deployment + CLI wrapper for MVP; Tauri desktop + npm SDK + Claude Code plugin form in v1.

### 6.4 Accessibility Constraints

- WCAG 2.1 AA compliance hardwired per Martinez Methods principle
- Additional standards: none specifically mandated; AAA criteria adopted where they don't add disproportionate complexity
- Target screen reader combinations: NVDA + Firefox, JAWS + Edge, VoiceOver + Safari

## 7. Assumptions

**Assumption 1: D2R plan format has a stable schema.**
- Rationale: the tool parses D2R plans per the D2R skill's specification
- Invalidated by: breaking changes to D2R plan format
- Fallback: schema versioning; tool supports multiple schema versions

**Assumption 2: pydantic/genai-prices maintains reasonable currency.**
- Rationale: primary pricing data source
- Invalidated by: abandonment or severe data quality degradation
- Fallback: direct scraping from provider pages as backup; manual JSON maintenance as final fallback

**Assumption 3: Opus 4.7 tokenizer correction factor is documentable.**
- Rationale: we need to apply a correction factor for accurate comparisons
- Invalidated by: Anthropic not publishing tokenizer details
- Fallback: Stage 00 includes empirical measurement of Opus 4.7 token ratio on a standard test corpus; derive correction factor from measurement

**Assumption 4: Users can produce D2R plans as input.**
- Rationale: the input format is a D2R plan
- Invalidated by: practitioners not adopting D2R
- Fallback: v0.2 adds a plan-authoring UI; v1.0 accepts simpler pipeline descriptions that the tool converts to D2R format

## 8. Open Questions

- **Open-source license selection.** MIT vs Apache 2.0 vs custom license protective of small users. Deferred to pre-launch legal review.
- **Pricing data update cadence.** CI pull at build time is baseline; consider per-week automation for rapid provider changes.
- **Post-run log format specification.** Each provider emits different response metadata formats. Need adapter layer for Anthropic, OpenAI, Google response formats.
- **Hosting domain.** `claudecost.io`, `cost.martinez-methods.com`, subdomain of `krystalmartinez.com`, or other — deferred to pre-launch.
- **Plugin distribution target.** Official Anthropic claude-plugins-official marketplace vs. independent distribution — deferred to v1.0.

## 9. Out Of Scope (Deferred)

Deferred to v0.2:
- Plan authoring UI (MVP requires users paste/upload existing plans)
- Retry scenario combinatorial modeling (MVP shows envelope only)
- XLSX export (MVP: CSV, JSON, Markdown)
- Plan library / persistence

Deferred to v1.0:
- Jupyter notebook export via Pyodide
- Quarto / LaTeX export for publication
- Tauri desktop shell
- npm SDK (engine as standalone package)
- Claude Code plugin form
- Pricing change history tracking
- Reproducibility package generator

Out of scope entirely for this product line:
- Cost optimization (auto-rewrite plans to minimize cost)
- Runtime pipeline cost monitoring
- Billing / payment processing
- Multi-user / team features
