## 2. Testing Taxonomy Applicability

### 2.1 Test Categories (Part 1, 20 Categories)

**Unit Tests**
- Applies: YES
- Exit criteria: 100% line + branch coverage of the cost calculation engine (`engine/`), parser (`parser/`), and export pipelines (`export/`). Every exported function has ≥ 1 unit test. Property-based tests via `fast-check` for cost math invariants (see category 16). Coverage enforced in CI via Vitest `--coverage` with failure on regression.

**Integration Tests**
- Applies: YES
- Exit criteria: parser → engine → export pipeline covered end-to-end with fixture D2R plans. Multi-plan comparison pipeline (parse N plans → compute deltas → render comparison data structure) covered. Post-run variance pipeline (estimate + execution log → variance report) covered. ≥ 20 integration test scenarios.

**Component / Module Tests**
- Applies: YES
- Exit criteria: every Svelte component has a component test using `@testing-library/svelte` + Vitest. Tests cover: rendering, user interaction, accessibility attributes, keyboard handling, reactive state. 100% coverage of components.

**End-to-End (E2E) Tests**
- Applies: YES
- Exit criteria: each user journey from PRD Section 4 has ≥ 1 E2E test via Playwright. Specifically: upload plan → view estimate → export (Journey 1), compare 3 plans (Journey 2), upload log → view variance (Journey 3), export Jupyter/CSV (Journey 4). Edge case tests: malformed plan rejection (Journey 5), stale pricing warning (Journey 6), tokenizer correction callout (Journey 7).

**Functional Tests**
- Applies: YES
- Exit criteria: every FR from TRD Section 2.1 has ≥ 1 test asserting its acceptance criteria. Traceability matrix maintained in `docs/testing/traceability.md`.

**Regression Tests**
- Applies: YES
- Exit criteria: full test suite runs on every PR via GitHub Actions; main branch protected; no merge without green build.

**Smoke Tests**
- Applies: YES
- Exit criteria: post-deploy health check runs basic "app loads, parses sample plan, computes estimate, renders result" within 30 seconds of deploy.

**Sanity Tests**
- Applies: YES
- Exit criteria: per-feature smoke-equivalent tests run on feature-branch builds.

**Acceptance Tests (UAT)**
- Applies: YES
- Exit criteria: Krystal executes the 4 primary user journeys from PRD Section 4 and confirms outcomes match expected. Documented in `docs/testing/uat-log.md` per release.

**Contract Tests (API-level)**
- Applies: NO
- Skip reason: pure client-side architecture; no backend API contracts. The pricing data source is a bundled JSON, not a runtime API. If v1 adds server-side PDF export, contract tests become applicable for that route specifically.

**Performance Tests**
- Applies: YES
- Exit criteria: Core Web Vitals at target (LCP ≤ 2.5s, FID ≤ 100ms, CLS ≤ 0.1) measured via Lighthouse CI. Plan parse ≤ 200ms for 100KB plans. Cost estimation ≤ 500ms for 20-stage plans. Multi-plan comparison ≤ 2s for 11-plan factorial. Bundle size ≤ 250KB gzipped. Enforced in CI.

**Security Tests**
- Applies: YES
- Exit criteria: `npm audit` zero high/critical vulnerabilities; Semgrep static analysis zero high-severity findings; dependency SBOM generated; strict CSP enforced on deployed site (no `unsafe-eval`, `unsafe-inline` only where unavoidable with justification). Verified via CI.

**Usability Tests**
- Applies: YES
- Exit criteria: Krystal performs manual usability testing simulating target user segments (researcher, non-coder practitioner). Documented findings in `docs/testing/usability-log.md`. Blockers resolved before release.

**Accessibility Tests**
- Applies: YES — HARDWIRED PER D2R
- Exit criteria: see Section 6 (dedicated accessibility criteria)

**Snapshot / Visual Regression Tests**
- Applies: YES
- Exit criteria: Playwright `toMatchScreenshot()` baseline for each chart type at reference data fixtures. CI fails on unexpected visual changes. Baselines updated only via explicit review.

**Property-Based / Fuzz Tests**
- Applies: YES
- Exit criteria: `fast-check` property tests for cost calculation invariants:
  - `totalCost(plan) ≥ Σ(stageCost(stage))` for all plans
  - `cost(plan with batch) ≤ cost(plan without batch)` always
  - `cost(plan with cache hit) ≤ cost(plan without cache)` always
  - `cost(retry scenario) ≥ cost(no-retry scenario)` for the same plan
  - `cost(opus4.7 plan) ≥ cost(opus4.6 plan)` for equivalent logical pipelines (tokenizer correction verified)
  - Cost never negative for any valid plan
  - Round-trip parse → serialize preserves semantics
  - 1000 generated examples per property

**Mutation Testing**
- Applies: YES (v0.2 — deferred but documented)
- Exit criteria: Stryker Mutator achieves ≥ 80% mutation score on the cost calculation engine. Deferred to v0.2 because v1.0 prioritizes shipping; adding mutation testing extends timeline.

**CI/CD Pipeline Tests**
- Applies: YES
- Exit criteria: GitHub Actions workflow runs on every push and PR: format → lint → typecheck → unit tests → integration tests → build → E2E tests → Lighthouse CI → axe-core → security audit. All jobs must pass for merge.

**Canary / A/B Testing**
- Applies: NO
- Skip reason: static site with single deploy target; no A/B infrastructure needed for MVP. Reconsidered if v1+ introduces multi-variant features.

**Monitoring / Observability**
- Applies: NO
- Skip reason: pure client-side with no server; no runtime metrics to capture in MVP. Future: optional opt-in anonymous usage counters in v1 (deferred with privacy review).

## 3. Standards Operationalized As Exit Criteria

### 3.1 Per-Standard Exit Criteria

**WCAG 2.1 AA**
- 1.1.1 Non-text Content: every non-decorative image has `alt` or `aria-label`. Tool: axe-core. Pass: zero violations.
- 1.3.1 Info and Relationships: semantic HTML used; ARIA roles where semantic HTML insufficient. Tool: axe-core + manual. Pass: zero axe violations + manual verification.
- 1.4.3 Contrast (Minimum): normal text ≥ 4.5:1, large text ≥ 3:1. Tool: axe-core. Pass: zero violations.
- 1.4.11 Non-text Contrast: UI components ≥ 3:1 contrast. Tool: axe-core. Pass: zero violations.
- 2.1.1 Keyboard: every function operable via keyboard. Tool: manual keyboard-only testing + axe. Pass: every interactive element reachable and operable.
- 2.1.2 No Keyboard Trap: no element traps keyboard focus. Tool: manual. Pass: focus escapes all components.
- 2.4.3 Focus Order: logical focus sequence. Tool: manual. Pass: focus progresses left-to-right, top-to-bottom, with logical skipping.
- 2.4.4 Link Purpose (In Context): link text describes destination. Tool: manual. Pass: no "click here" or "learn more" without context.
- 2.4.7 Focus Visible: visible focus indicator on all interactive elements. Tool: manual. Pass: `focus-visible` styles applied everywhere.
- 3.1.1 Language of Page: `lang` attribute set. Tool: axe-core. Pass: `lang="en"` (or applicable) on every page.
- 3.2.1 On Focus: focus does not trigger context change. Tool: manual. Pass: focus never triggers nav or state change.
- 3.3.1 Error Identification: errors identified in text. Tool: manual + axe. Pass: every error has a visible text explanation.
- 4.1.2 Name, Role, Value: all UI components have accessible name, role, and current state exposed to AT. Tool: axe-core + screen reader manual. Pass: zero axe violations + manual SR verification.

**OWASP Top 10 (2021)**
- A03 Injection: plan parser uses Zod validation; no dynamic eval of plan content. Verified by code review + input fuzzing (category 19).
- A05 Security Misconfiguration: strict CSP enforced, HSTS with preload, no `X-Powered-By`. Verified by `securityheaders.com` scan: A+ rating required.
- A06 Vulnerable Components: `npm audit` on every CI run; zero high/critical findings.
- A08 Data Integrity Failures: SRI (Subresource Integrity) on all external scripts (none expected in MVP).

**CERT Secure Coding (TypeScript)**
- No `any` types without explicit justification comment
- No dynamic evaluation of untrusted input
- All external data validated via Zod before use
- Verified by ESLint strict + typecheck.

### 3.2 Regulatory Compliance Exit Criteria

- No personal data processed → no regulatory compliance exit criteria triggered. Documented in privacy statement.

## 6. Accessibility Criteria (Detailed)

### 6.1 Target Compliance

- Minimum: WCAG 2.1 AA
- Additional AAA criteria adopted: 1.4.6 Contrast (Enhanced) where practical, 2.3.3 Animation from Interactions

### 6.2 Automated Testing

- Tool: `@axe-core/playwright` integrated into Playwright E2E suite
- CI integration: `critical` or `serious` violations FAIL the build
- Scope: every page, every component state variant, every interactive flow

### 6.3 Manual Testing Protocol

Per release:
- Screen reader testing on 3 combinations: NVDA+Firefox, JAWS+Edge, VoiceOver+Safari
- Keyboard-only navigation through every user journey
- Color contrast spot-check on dynamic-color elements (charts, callouts)
- `prefers-reduced-motion` verification
- 200% browser zoom — no loss of functionality or content

Documented in `docs/testing/a11y-manual-log.md`.

### 6.4 Accessibility Exit Criteria For Every UI Stage

- Zero axe-core `critical` or `serious` violations
- All functionality keyboard-operable
- All interactive elements have visible focus indicators via `focus-visible:ring-2 focus-visible:ring-[var(--color-accent)]`
- All images have alt text or explicit `aria-hidden="true"` for decorative
- Color never the sole indicator of meaning (charts have data tables; status indicators have icons + text)
- Contrast ≥ 4.5:1 (normal), ≥ 3:1 (large), ≥ 3:1 (UI components)
- Touch targets ≥ 44x44px (web minimum enforced)
- `prefers-reduced-motion` respected on all animations
