## 2. System Shape

### 2.1 One-Paragraph System Description

Claude Cost is a pure client-side TypeScript application that parses structured D2R code plans, calculates cost estimates against a bundled multi-provider pricing database, analyzes post-run actual-vs-estimated variance, and exports results in multiple formats. A shared calculation engine serves the web UI (SvelteKit static site), the CLI wrapper (Node.js), and in v1 the Tauri desktop surface, npm SDK, and Claude Code plugin form. No backend server; no data transmission; all computation happens in the user's browser or local CLI process.

### 2.2 Architectural Style

- **Static site + CLI wrapper (MVP):** web surface is a static SvelteKit build deployed to a CDN; CLI wrapper is a Node.js binary consuming the same engine
- **Library-plus-consumers (v1):** core engine becomes a standalone npm package with multiple consumer surfaces
- **Synchronous client-side:** all computation is synchronous from the user's perspective; no async backend calls during normal operation (exception: one serverless edge function for PDF export in v1)
- **Stateless:** no persistent state across sessions; results can be saved to files but the app itself holds no long-lived state
- **Pure client-side:** no server-side rendering; no backend API for core functionality

### 2.3 Surface Layers

**MVP Surfaces:**

| Surface | Consumer Type | Relationship To Others |
|---------|---------------|------------------------|
| Web app | Non-coder researchers, practitioners, non-technical users | Wraps core engine with SvelteKit UI |
| CLI | Krystal (experimental workflow), developers, scripted pipelines | Wraps core engine with Commander.js-style CLI |

**v1 Additional Surfaces:**

| Surface | Consumer Type | Relationship To Others |
|---------|---------------|------------------------|
| Tauri desktop | Privacy-sensitive users; offline use | Wraps web UI with local execution |
| npm SDK | Plugin authors; integration with other tools | Core engine as standalone package |
| Claude Code plugin | In-session cost estimation during Claude Code work | Consumes npm SDK |

All surfaces share the same core engine. The engine is the IP; surfaces are distribution channels.

## 3. Components And Boundaries

### 3.1 Component Inventory

**Core Engine (`engine/`)**
- Responsibility: pure TypeScript calculation logic for cost estimation, post-run variance analysis, and delta computation
- Inputs: validated D2RPlan objects, PricingModel database
- Outputs: EstimateResult, VarianceReport, ComparisonReport
- Interfaces: pure functions; no side effects; no I/O

**Plan Parser (`parser/`)**
- Responsibility: parse YAML or Markdown D2R plan documents into validated D2RPlan internal representations
- Inputs: raw text (YAML or Markdown)
- Outputs: parsed + Zod-validated D2RPlan object OR structured parse error
- Interfaces: single `parsePlan(rawText, format): Result<D2RPlan, ParseError>` function

**Pricing Database (`pricing/`)**
- Responsibility: provide current pricing data for all modeled providers and models
- Inputs: none at runtime (data bundled at build time)
- Outputs: PricingModel lookups by model name
- Interfaces: single `getPricing(modelId): PricingModel` function; list of available models
- Build-time input: `@pydantic/genai-prices` package + custom `anthropic_advanced_pricing.json`

**Execution Log Parser (`logs/`)**
- Responsibility: parse provider-specific API execution logs (Anthropic, OpenAI, Google response metadata formats) into normalized ExecutionLog representation
- Inputs: raw log JSON from any supported provider
- Outputs: normalized ExecutionLog OR parse error
- Interfaces: `parseLog(rawLog, provider): Result<ExecutionLog, ParseError>`

**Export Pipeline (`export/`)**
- Responsibility: convert engine outputs into exportable formats
- Inputs: EstimateResult / VarianceReport / ComparisonReport
- Outputs: CSV / JSON / Markdown report (MVP); + XLSX / Jupyter notebook / Quarto qmd (v1+)
- Interfaces: one function per target format

**Web UI Application (`web-ui/`)**
- Responsibility: SvelteKit-based UI providing all MVP user journeys through interactive browser interface
- Inputs: user interactions, engine outputs
- Outputs: rendered UI, triggered exports
- Interfaces: consumed by end-users

**CLI Application (`cli/`)**
- Responsibility: Node.js CLI wrapper for programmatic/scripted use
- Inputs: command-line arguments + file paths
- Outputs: JSON to stdout, files to specified paths, human-readable progress to stderr
- Interfaces: `claude-cost estimate`, `claude-cost analyze`, `claude-cost compare`, `claude-cost export`

**Visualization (`viz/`)**
- Responsibility: render charts and tables from engine outputs
- Inputs: EstimateResult / VarianceReport / ComparisonReport
- Outputs: Svelte components rendering ECharts visualizations with accompanying accessible data tables
- Used by: web UI only (CLI has text-based visualization for terminals)

### 3.2 Boundaries

**Process boundaries:**
- Web app runs entirely in the browser process; no external process communication
- CLI runs in a Node.js process; no sub-process spawning for core operations (exception: v1 may spawn Python subprocess for Jupyter export if Pyodide in-browser is unacceptably slow)

**Language boundaries:**
- MVP: 100% TypeScript (browser + Node.js)
- v1 additions: Pyodide WASM Python for Jupyter export (in-browser, no separate runtime); Rust via Tauri for desktop shell only

**Trust boundaries:**
- All user-uploaded content (plans, execution logs) treated as untrusted input → Zod validation at parse boundaries
- No cross-user trust (no multi-user system)
- Pricing data is trusted at build time (vetted source: pydantic/genai-prices) but pricing staleness is signaled at runtime

**Release boundaries:**
- MVP: web + CLI release together (same version); share npm workspace
- v1: web + CLI + npm SDK + Tauri desktop all release together; plugin form may release independently after core surfaces stable

## 4. Data Flow

### 4.1 Primary Data Flows

**Flow 1: Pre-Run Cost Estimation**
1. User provides D2RPlan (paste, upload, or CLI file path)
2. Parser validates and parses to internal D2RPlan
3. Engine looks up pricing per stage's model
4. Engine calculates per-stage cost with caching/batching/thinking-mode/tokenizer-correction factors applied
5. Engine produces EstimateResult (per-stage costs, total, envelope bounds, assumptions log)
6. Surface renders result (web UI: charts + tables; CLI: JSON to stdout)
- Latency expectation: < 500ms for 20-stage plan

**Flow 2: Multi-Plan Comparison**
1. User provides 2-20 D2RPlans
2. Parser validates each
3. Engine calculates per-plan EstimateResult
4. Engine produces ComparisonReport (deltas, heatmaps, concentration analysis)
5. Surface renders comparison view
- Latency expectation: < 2s for 11-plan comparison

**Flow 3: Post-Run Variance Analysis**
1. User provides ExecutionLog (JSON from provider API) + original EstimateResult (or original D2RPlan to regenerate)
2. Log parser normalizes to ExecutionLog
3. Engine pairs actuals with estimate per stage
4. Engine produces VarianceReport (per-stage variance, attribution, aggregate)
5. Surface renders variance report
- Latency expectation: < 1s for single-plan variance

**Flow 4: Export**
1. User triggers export from any result view
2. Surface invokes appropriate export pipeline function
3. Export pipeline converts result to target format
4. File downloaded (web) or written to disk (CLI)
- Latency expectation: < 1s for CSV/JSON; < 3s for Markdown report; v1: < 30s for Jupyter (Pyodide-mediated)

### 4.2 Secondary Data Flows

- Build-time pricing data update: CI script pulls from `@pydantic/genai-prices` → bundles into JSON → deploys with build
- No runtime background jobs

### 4.3 Data Persistence Points

- **None at MVP.** The web app holds state only for the duration of a browser session; the CLI holds state only for the duration of a command invocation. Results are exported to files at user request; the app does not manage those files after export.
- Browser localStorage may be used to remember UI preferences (theme, last-viewed plan) but NOT to persist user data (plans, execution logs). No PII or sensitive data persists anywhere.

## 5. Deployment Architecture

### 5.1 Deployment Targets

**Web static site (MVP):**
- Hosting: Vercel or Cloudflare Pages (free tier)
- Build: `npm run build` produces static assets
- Distribution: deployed to CDN; user loads via HTTPS
- Release cadence: continuous (on merge to main)

**CLI npm package (MVP):**
- Hosting: npm registry
- Build: `npm run build:cli` bundles Node-compatible output
- Distribution: `npm install -g @martinez-methods/claude-cost-cli`
- Release cadence: semver releases with changelog

**v1 additions (deferred):**
- Tauri desktop bundles: GitHub Releases, each platform (Windows/macOS/Linux)
- npm SDK package: `@martinez-methods/claude-cost-engine`
- Claude Code plugin: official plugin marketplace submission

### 5.2 Runtime Environments

**Web:**
- Firefox 115+, Chrome 120+, Safari 17+, Edge 120+
- Minimum: 2GB RAM, modern CPU (2020+)
- Network: required for initial load only; works offline after

**CLI:**
- Node.js 20+ LTS
- Linux, macOS, Windows
- No GUI required

**Tauri desktop (v1):**
- Windows 10+, macOS 11+, Ubuntu 22.04+
- 100MB disk for install

### 5.3 Configuration And Secrets

- **No secrets.** The product holds no secrets and requires no API keys to operate.
- Configuration is minimal: pricing data freshness date, UI theme preferences, default export format.
- Configuration delivered via bundled constants (pricing) + localStorage (user preferences).

### 5.4 Deployment Topology

- MVP: single static web deploy + single CLI npm package. No inter-target communication.
- v1: same independence; each surface consumes the shared engine at build time but runs independently at runtime.

## 6. Cross-Cutting Concerns

### 6.1 Logging And Observability

- MVP: console logging in development only; no production telemetry for privacy posture
- v1: may add opt-in anonymous usage counters (privacy-reviewed before implementation)

### 6.2 Error Handling Strategy

- **Result types throughout.** Functions that can fail return `Result<T, ErrorType>` rather than throwing. Forces error handling to be explicit.
- **Typed errors per boundary:** `ParseError` for parser; `PricingError` for pricing lookups; `ExecutionLogError` for log parsing.
- **User-facing error presentation:** every error has a typed description + suggested remediation. No raw stack traces or opaque error codes.
- **No error swallowing:** errors propagate to the surface and render to the user.

### 6.3 Concurrency Model

- Single-threaded event loop (standard browser / Node.js model)
- No worker threads in MVP (calculations are fast enough not to need them)
- v1: may add Web Workers for very large plan parsing if profiling shows need

### 6.4 Security Architecture

- **Zero server-side attack surface** (no backend)
- **Zero data-transmission attack surface** (no data transmitted to any server during normal use)
- Input validation at every external boundary (Zod on parse; file type validation on upload)
- Strict Content Security Policy for web surface
- No dynamic evaluation of user input (no `eval`, no `Function()` constructor)
- HTTPS only; HSTS preload eligibility

### 6.5 Accessibility Architecture

- **Hardwired WCAG 2.1 AA at component layer.** All UI primitives sourced from `shadcn-svelte` (Radix-based, accessibility-by-default).
- Automated testing: `@axe-core/playwright` integrated into CI; violations at `critical` or `serious` level fail the build.
- Manual testing protocol documented in TQCD Section 6.3.
- Every chart has an accessible data table alternative (per GOV.UK design system pattern).

## 7. Architectural Decisions (Mini-ADRs)

### AD-01: Pure client-side architecture (no backend server)

- Status: Accepted
- Context: The product is a calculator operating on structured public data (pricing) and user-provided structured data (plans). No multi-user coordination required. No cross-user data. No intrinsic need for a server.
- Options: (a) pure client-side, (b) thin API backend for pricing freshness, (c) full backend with user accounts and persistence
- Decision: (a) pure client-side
- Rationale: Eliminates entire classes of security + privacy + operational complexity. User's proprietary D2R plans never leave their machine. No infrastructure to maintain. Deployment is trivial. Trade-off: limits certain features (e.g., shared plan libraries across users, cloud sync) that are NOT MVP requirements.
- Consequences: Commit to browser-native + Node-native libraries throughout. No server-dependent features (search over plans by other users, etc.) possible without re-architecture.

### AD-02: SvelteKit 5 + Svelte 5 runes for web UI

- Status: Accepted
- Context: Need a modern, accessible, performant web UI framework for a data-dense tool targeting mid-range hardware and non-coder users.
- Options: (a) SvelteKit 5 + Svelte 5 runes, (b) React + Next.js, (c) SolidStart, (d) Astro with islands
- Decision: (a) SvelteKit 5 + Svelte 5 runes
- Rationale: Bundle size ~42KB vs Next.js ~120KB; compile-time accessibility warnings; runes provide fine-grained reactivity for cost-recalculation-on-input; ecosystem mature in 2026; strong AI-assisted coding support.
- Consequences: Lock-in to Svelte ecosystem for UI components; must choose Svelte-compatible viz library (drove ECharts with Svelte wrapper selection).

### AD-03: Apache ECharts + companion data tables for visualization

- Status: Accepted
- Context: Need bar charts, stacked bars, waterfall, heatmap, comparison tables. Accessibility required. SvelteKit-compatible.
- Options: (a) Apache ECharts, (b) Plotly.js, (c) D3.js raw, (d) Observable Plot
- Decision: (a) Apache ECharts with `role="img"` + `aria-labelledby` + companion accessible data tables
- Rationale: Handles all required chart types including waterfall via custom series; good performance; Svelte wrapper available; ARIA story workable with companion-table pattern.
- Consequences: Every chart must ship with a data table alternative (WCAG 1.1.1 compliance path). Accepts known keyboard-navigation gap in ECharts itself, compensated by the companion table.

### AD-04: Shared core engine as standalone package (v1)

- Status: Accepted (v1 scope)
- Context: Multiple surfaces (web, CLI, Tauri, plugin) need the same calculation logic.
- Options: (a) shared engine as npm package, (b) duplicate engine per surface, (c) engine in a server consumed by all
- Decision: (a) shared engine as npm package (`@martinez-methods/claude-cost-engine`)
- Rationale: Single source of truth for calculation logic; cleanly reusable; enables independent versioning.
- Consequences: Requires maintaining versioned API boundary in engine package. Changes to engine propagate to all consumers via version bumps.

### AD-05: Pydantic genai-prices as primary pricing source

- Status: Accepted
- Context: Need current pricing across providers. Must be reliable and current.
- Options: (a) @pydantic/genai-prices npm package, (b) direct scraping, (c) manual JSON maintenance only, (d) third-party API
- Decision: (a) primary + (c) secondary for Anthropic-specific advanced pricing
- Rationale: pydantic/genai-prices is the most comprehensive open-source pricing database in 2026 (1000+ models, 30+ providers, active maintenance, npm-packaged). Manual Anthropic overlay handles cache tiers, fast mode, thinking-mode specifics not in base library.
- Consequences: Dependency on upstream maintainer cadence. Mitigated by build-time bundling (no runtime dependency) and manual overlay for Anthropic specifics.

## 8. Technical Debt And Known Compromises

**Compromise 1: ECharts keyboard accessibility gap**
- Not bypassable in ECharts core; companion data table provides WCAG 2.1 AA compliance path but does not provide keyboard interaction with the chart itself
- When to revisit: if WCAG 3.0 enforces stricter standards, re-evaluate visualization library

**Compromise 2: Pricing data staleness tolerance**
- Bundle-at-build-time means pricing can be up to 30 days stale before warning
- Warning is visible but does not block operation
- When to revisit: if pricing change velocity increases materially, consider weekly rebuild automation

**Compromise 3: Thinking-mode token prediction**
- Cannot predict actual thinking-mode token consumption (it's output-dependent)
- Tool reports "up to X thinking tokens" rather than predicted value
- Acknowledged as fundamental product limitation; user-visible

**Compromise 4: Pyodide Jupyter export size (v1)**
- Pyodide WASM adds ~15MB to load if user requests Jupyter export
- Lazy-loaded so doesn't affect initial load
- Alternative (serverless Python function) rejected due to architectural preference for pure client-side
