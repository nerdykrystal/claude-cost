## The Origin (Age 2)

The Bobo Framework was installed by Krystal's parents when she was two years old. When Krystal did something wrong, the mechanism was:

> **"Sorry didn't do it. Krystal did it. So what's Krystal going to do to fix it?"**

The framework has five steps:

1. **Make expectations explicit.**
2. **The one who erred must own the error.**
3. **The one who erred is responsible for the solution.**
4. **The one who erred is responsible for changed behavior.**
5. **Apology alone is never accepted — what are you going to do differently?**

This is the correction-to-rule pipeline at its origin. Not a professional methodology — a survival methodology. Krystal's parents — her mother (a Puerto Rican immigrant who came to the mainland at 17 speaking no English) and her father (a high school dropout who worked multiple simultaneous jobs) — could not navigate the Liverpool suburbs they moved their daughter into at age 6. What they gave her instead was this methodology: try the thing with no safety net, explicitly ask for feedback, reject praise as "real" feedback, take action on the feedback, treat the chance to improve as a blessing.

## Why Bobo Applies Natively to AI

AI systems produce observable errors. Errors can be classified. Classified errors can produce rules. Rules can be enforced structurally (prompt, hook, test, gate). The pattern maps one-to-one.

Traditional AI response to error: retrain, tune, prompt-engineer empirically until improved.
Bobo-framework response to error: articulate the specific mechanism, produce a specific rule, enforce that rule STRUCTURALLY (not through model behavior but through non-bypassable infrastructure around the model).

This is why Martinez Methods produces artifacts like:
- **ASAE Certainty Threshold** — rule: N consecutive clean passes required; enforced structurally via convergence gate
- **CDCC non-bypassable hooks** — rule: sub-agent must execute tests before returning; enforced structurally via `PostToolUse` hook refusing non-compliant calls
- **D2R 4-document prerequisite** — rule: PRD + TRD + AVD + TQCD required before Stage 00; enforced structurally via skill that refuses to proceed without them
- **Two-Tier Verification Gate** — rule: HIGH severity ≤5%, TOTAL ≤10%; enforced structurally via deterministic gate with binary pass/fail
- **Unanimous Agreement Audit** — rule: 100% agreement triggers 100% external verification; enforced structurally via mandatory external-source check outside thread evidence chain

Every one of these is a Bobo instance. AI fails → Krystal articulates the mechanism → rule produced → rule enforced structurally.

## Recursive Application #1: Sonnet Documentation Failures → ASAE Genesis

The first AI-substrate Bobo cycle produced ASAE (ASAE Certainty Threshold) in ~2026-02. Canonical origin at `~/.claude/projects/C--Users-NerdyKrystal-repos/memory/project_asae_origin.md`. Progressive interventions:

1. "Here are the specs again" — Sonnet tried again, different errors. *Result: telling the model what to do twice doesn't fix the checking problem.*
2. "Go back and check your work" — Sonnet revised, caught more errors, but still not great. *Result: verifiers-as-auditors help but don't converge.*
3. "Audit your work against the spec" — output SO MUCH BETTER. *Key insight: the model needs an explicit reference spec to audit against.*
4. Print → page 5 failure — Sonnet forgot instructions for ~2/3 of a section then hopped back on track. *Result: one audit pass is not enough.*
5. "Audit against the spec again" — previously-caught issues fixed, but a NEW error appeared on page 6-7. *Result: single re-audit catches known errors but misses new ones.*
6. **The breaking point** — "Don't you DARE declare you're done after one pass. Audit, re-audit, re-re-audit. Make changes between audits. And I don't trust YOUR judgment that you're done — you ctrl+F and never read to the end. So check and recheck until you are CERTAIN, which means at LEAST TEN separate passes all returning zero errors."

This is the Bobo Framework producing the ASAE Certainty Threshold rule. Five steps:
1. **Expectations explicit** — specs provided
2. **Ownership** — Sonnet must own the errors, not Krystal
3. **Responsibility for solution** — Sonnet must remediate, not just acknowledge
4. **Responsibility for changed behavior** — the rule isn't "don't do this again"; it's "audit N consecutive times to a stable zero-error state"
5. **No apology alone** — the structural enforcement is convergence-to-stable-state across consecutive independent passes

ASAE is a Bobo rule encoded structurally. The rule survives the specific Sonnet-documentation context that produced it and generalizes to any domain (code, docs, research, legal, instructional) via domain-specific adapters.

**Ink-cost constraint context (testimony-only):** Krystal's direct statement in this session: "i was printing the sonnet outputs and ink is very expensive so i needed it to keep remediating its own audits until what i was giving feedback on when printed was all things that needed my input." No committed primary source connects the ink-cost driver to the convergence-requirement in prose. The Best Practices §2.10 documents the handwritten-annotation workflow as of 2026-02-27 but doesn't name ink cost. **This is the direct Franklin parallel** — tradesman's material cost (ink) drives iteration discipline that becomes methodology. ### Rule produced

```
Phase 1 — Question [N] ([label]): insufficient.
Reason: [specific diagnosis — what the answer lacks].
Next-step questions:
- [specific follow-up 1]
- [specific follow-up 2]
- [specific follow-up 3]

This skill will not proceed to PRD authorship until Question [N] is answered adequately.
```

The fail-condition format is itself a Bobo instance at the skill-invocation level: error articulated with specific diagnosis → next-step questions produced as the correction path → skill refuses to proceed = structural enforcement.

**Bobo five-step applied:**

1. **Expectations explicit** — the 5 interrogation questions with worked examples of pass/fail answers
2. **Ownership** — "This skill will not proceed to PRD authorship until Question [N] is answered adequately" forces user to own the ideation gap
3. **Responsibility for solution** — next-step questions push the user to produce the missing specificity themselves
